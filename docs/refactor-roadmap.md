# shuqian-nav 重构路线图

## 目标

本次重构不是简单换皮，而是把项目拆成更清晰的两个产品面：

- 首页是公开导航站，只负责浏览、搜索、分类筛选和打开书签。
- 后台是受保护的管理系统，只负责书签、分类、导入、回收站、链接检查、令牌、通知、备份和密码等管理动作。

最终效果应该是：用户打开首页不需要密码，管理员登录后进入一个统一、清晰、速度更快的后台。代码上首页和后台互不加载彼此的业务脚本，只共享认证、API 客户端、基础样式变量和统一头部。

## 技术栈选择

### 推荐结论

短期继续使用当前技术栈：Cloudflare Pages + Pages Functions + D1 + 原生 HTML/CSS/JS。

原因：

- 项目已经部署在 Cloudflare Pages，后端 Functions 和 D1 绑定都围绕这个模式运行。
- 当前没有构建步骤，部署流程简单，GitHub Actions 已经覆盖测试、D1 初始化和 Pages 部署。
- 功能规模还没有大到必须引入 React/Vue/Svelte；现在最大的问题是边界混乱、页面重复、样式不统一，不是框架能力不足。
- 用户端首页需要快，原生静态页面更容易做到低依赖、低首屏成本。

### 不建议现在做的事

- 不拆成两个 Cloudflare Pages 项目。首页和后台仍应同源，避免登录跳转、API CORS、部署变量和 D1 绑定复杂化。
- 不立即引入 Vite/React/Vue。框架迁移会放大改动范围，先把模块边界和 UI 规范稳定下来。
- 不把所有脚本改成 ES Modules。当前 `window.X` 单例模式虽然朴素，但和无构建部署匹配；要迁移也应放到第二阶段。

### 可以后续评估的升级

当后台页面继续增多、状态交互明显复杂后，可以再评估：

- Vite + TypeScript + Lit：保留 Web Components 思路，侵入性较低。
- Vite + React：后台表格、批量操作、弹窗状态会更容易维护，但需要一次较大迁移。
- Tailwind 不作为首选。项目已有较多 CSS，继续建立本地设计 token 和组件类更稳。

### 后端边界

API 分三类：

- 公开读取：`GET /api/bookmarks`、`GET /api/bookmarks/categories`、单个书签读取、访问统计上报。
- 管理写入：新增、编辑、删除、批量移动、分类 CRUD、导入、清空、回收站恢复/永久删除。
- 系统管理：链接检查、备份、迁移、性能、通知、令牌、密码。

要求：

- 公开读取接口不能依赖登录态。
- 所有写入和系统管理接口必须走 `authenticateRequest` 或统一中间件。
- API 响应统一使用 `ResponseHelper`，避免每个 handler 手写不同格式。
- CORS 和安全头只在 `_middleware.js` 或指定公共层处理，单个 API 不重复写跨域头。

## 首页重构

首页定位：公开导航，不是后台入口页。

需要保留：

- 搜索。
- 分类筛选。
- 排序。
- 书签卡片。
- 打开链接和访问统计。
- 未登录状态下显示“登录后台”入口。

需要移出首页或登录后才显示：

- 新增书签。
- 编辑/删除按钮。
- 导入。
- 备份。
- 修改密码。
- 链接检查。
- 回收站。
- 令牌管理。
- 通知管理。

首页 UI 方向：

- 更像导航站，而不是后台面板。
- 首屏重点放搜索、分类和书签内容，不放大面积设置入口。
- 卡片图片或 favicon 需要清晰，统一尺寸，避免模糊和布局跳动。
- 未分类显示要准确：优先使用后端返回的 `category_name`，没有分类才显示“未分类”。
- 空状态要给普通用户可理解的信息，不能出现后台术语。

## 后台重构

后台定位：受保护的工作台。

### 菜单重构

后台菜单应该有稳定的信息架构：

- 概览：后台首页、统计、最近问题。
- 内容管理：书签管理、分类管理、回收站。
- 数据工具：导入、导出、备份。
- 维护工具：链接检查、通知、性能。
- 账号与安全：修改密码、同步令牌、登出。

要求：

- 顶部只保留当前页面标题、主要动作和账户入口。
- 左侧或顶部下拉菜单只能有一套，不要页面内再重复一套相同入口。
- 管理页按钮按任务出现，不要所有页面都放“导入/令牌/链接检查/回收站”。
- 菜单 active 状态由 `data-page-key` 或统一路由映射控制。

### 页面重构

优先级：

1. 书签管理：搜索、分类筛选、多选、批量移动、批量删除、分页、单条编辑。
2. 分类管理：新建、编辑、颜色、描述、书签数量、删除迁移目标。
3. 链接检查：清晰区分可访问、疑似被拦截、明确不可访问；删除只针对明确异常。
4. 导入：导入前预览、替换导入强确认、一键清空强确认。
5. 回收站：批量恢复、批量永久删除、按原因/时间筛选。
6. 通知和令牌：保留功能，重做成统一后台样式。

### 管理页 UI 方向

- 使用统一后台布局：导航区、标题区、工具条、内容区、分页/状态区。
- 表格和列表以密度、可扫描性为主，不做营销式大卡片。
- 所有危险操作使用统一确认弹窗，避免 `confirm()`。
- 所有加载、空状态、错误、成功提示使用统一组件。
- 移动端可以折叠菜单，但管理页优先保证桌面可用性。

## 样式系统

第一阶段不引入第三方 UI 库，建立本地设计系统。

必须沉淀：

- 颜色 token：背景、文本、边框、主色、危险、警告、成功。
- 间距 token：4、8、12、16、24、32。
- 字号层级：首页标题、后台标题、卡片标题、表格正文、辅助文字。
- 组件类：按钮、图标按钮、输入框、选择框、标签、表格、分页、弹窗、toast、空状态。

文件建议：

- `public/css/tokens.css`：变量。
- `public/css/components.css`：按钮、表单、弹窗、toast、菜单等组件。
- `public/css/home.css`：首页专属布局和卡片。
- `public/css/admin.css`：后台壳、表格、工具条。

## 多 agent 工作方式

本项目适合 coordinator + 多 agent 并行推进。每个 agent 只负责一个边界，最后由主线程评审和合并。

### Agent 分工

| Agent            | 任务                         | 主要产物                                   | 验收                                   |
| ---------------- | ---------------------------- | ------------------------------------------ | -------------------------------------- |
| Agent A 架构     | 梳理目录、依赖、页面加载关系 | 目录重组 PR、加载矩阵                      | 首页不加载后台脚本，后台不加载首页脚本 |
| Agent B 首页     | 首页 UI 和公开浏览体验       | `home.css`、首页脚本整理                   | 未登录可浏览，卡片清晰，搜索/分类正常  |
| Agent C 后台菜单 | 管理菜单和后台壳             | `admin.css`、`site-menu.js` 或 admin shell | 菜单不重复，active 正确，后台页一致    |
| Agent D 管理页面 | 书签/分类/回收站/导入页面    | 页面模块和统一组件                         | 批量操作、确认弹窗、分页可用           |
| Agent E API/安全 | 接口权限和响应统一           | API 权限清单、测试                         | 公开 GET 免登录，写接口 401，测试覆盖  |
| Agent F QA       | 浏览器回归和性能             | 测试脚本、截图、问题清单                   | 首页/后台关键流程通过                  |

### 协作规则

- 每个 agent 使用独立分支或 worktree，不直接改 main。
- 每个 agent 的改动必须小而完整：能跑测试，有清楚总结。
- 主线程只合并通过验收的结果，不把多个半成品直接揉到 main。
- UI agent 必须提供桌面和移动端截图或 Playwright/Puppeteer 验证输出。
- API agent 必须补 `tests/api-core.test.js`。
- 如果两个 agent 都要改 `site-menu.js`，先由架构 agent 定接口，再由 UI agent 改样式。

### 推荐执行顺序

1. 建立重构基线：跑 `npm test`，记录当前首页和后台截图。
2. Agent A 做目录和加载矩阵，不改功能。
3. Agent E 固化权限边界和 API 测试。
4. Agent B 重构首页 UI。
5. Agent C 重构后台菜单和后台 shell。
6. Agent D 分页改造管理页面。
7. Agent F 做回归、性能和视觉检查。
8. 主线程合并、格式化、测试、推送、观察 GitHub Actions。

## 验收清单

### 首页

- 未登录访问 `/` 不跳登录。
- 首页只请求公开读取接口。
- 书签、分类、搜索、排序、分页或加载更多正常。
- 未登录不显示编辑/删除等管理按钮。
- 点击新增、管理菜单、后台链接会进入登录流程。
- favicon/图片清晰，不模糊，不撑破卡片。

### 后台

- 未登录访问任何后台页都会跳登录。
- 登录后后台菜单统一，无重复按钮。
- 书签管理支持搜索、筛选、多选、批量移动、删除。
- 分类管理支持创建、编辑、删除迁移。
- 危险操作全部使用统一确认弹窗。
- 页面加载失败时有明确错误提示。

### API

- 公开读取接口 200。
- 所有写接口无 token 返回 401。
- 链接检查不会把“被拦截但实际可访问”的站点当作明确删除对象。
- 密码修改仍优先使用 D1 中的非默认密码，环境变量可作为部署初始配置。

### 质量

- `node --check` 覆盖改动 JS。
- `npm test` 通过。
- Prettier 和 ESLint 通过。
- 浏览器验证桌面和移动端关键页面。
- 推送后 GitHub Actions 成功。

## 风险和处理

- 风险：一次性大改导致线上不可用。处理：按页面和边界拆分，每次小步提交。
- 风险：公开首页误开放写接口。处理：API 测试强制覆盖 401。
- 风险：菜单继续重复。处理：所有入口只从 `site-menu.js` 或后台 shell 渲染。
- 风险：无构建项目缓存导致 UI 不更新。处理：每次改 JS/CSS 更新 query version。
- 风险：多 agent 改同一文件冲突。处理：先定义接口，再分头做页面。
