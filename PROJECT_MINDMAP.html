shuqian-nav 项目脑图

这是基于仓库 rdone4425/shuqian-nav 整理的项目脑图。 我按“产品定位、前端页面、后端 API、数据库、认证、AI 能力、部署运维”几个视角来拆，适合你自己回看项目结构，也适合拿给别人快速理解。

定位Cloudflare Pages + Functions + D1 的书签导航后台 前端public/ 多页面静态站点 后端pages/functions/ API + 中间件 + utils 数据db/schema.sql 单一数据库结构源

项目总览

先从最高层看，这个项目并不是单纯的书签列表，而是一个带登录、分类、导入导出、回收站、链接检查、通知、Token、AI Wiki 的完整小型后台系统。

shuqian-nav
一个部署在 Cloudflare Pages 上的书签导航系统。 静态页面放在 public/，API 放在 pages/functions/， 数据库使用 D1，部署链路由 GitHub Actions 自动完成。
01产品定位
核心不是“导航页”，而是“有后台能力的书签管理系统”。
基础能力
书签增删改查、分类、搜索、排序、收藏、访问统计。
管理能力
登录、改密码、Token、导入导出、回收站、配置管理。
增强能力
链接检查、通知、AI 设置、AI Wiki 快照、自动备份。
02前端页面层
public/ 是一个多页面静态应用，每个页面对应一个管理功能。
index.html
主书签导航页，核心交互入口。
login.html
后台登录入口。
功能页集合
import.html、deleted-bookmarks.html、link-checker.html、notifications.html、token.html、ai-settings.html
共用资源
public/css/、public/components/header.html、public/_headers、public/_redirects
03前端 JS 模块
前端代码不是框架项目，而是原生 JS 模块化组织。
public/js/app.js
主应用控制器，负责初始化、事件绑定、书签页交互和设置面板逻辑。
public/js/auth.js
认证状态、登录登出、token 校验、重定向和权限事件。
public/js/wiki-view.js
Wiki 风格知识库视图，可切换本地视图和 AI 视图。
其他模块
bookmarks.js、import.js、deleted-bookmarks.js、i18n.js、ui-enhancements.js、utils/*.js
04后端 API 层
pages/functions/ 是项目真正的业务后端，按 Pages Functions 路由约定组织。
_middleware.js
全局中间件，处理 CORS、基础安全头、可疑请求拦截。
认证 API
/api/auth/login、verify、change-password、token
书签 API
/api/bookmarks、/api/bookmarks/[id]、categories、import、sync、deleted、keep-status、visit
系统 / 扩展 API
/api/system/*、/api/wiki、/api/analytics、/api/cron/weekly-check、/api/health
05后端工具层
很多公共能力被下沉到 pages/functions/utils/，这是后端的复用层。
database-helper.js
统一 D1 查询、分页、批量执行、增删改查辅助。
auth-middleware.js / jwt-manager.js
认证包装、JWT secret 处理和鉴权统一入口。
response-helper.js
成功、校验失败、未授权、服务端错误等统一响应格式。
业务工具
backup-manager.js、bookmark-analytics.js、deleted-bookmarks.js、performance-monitor.js、validation.js、ai-client.js、ai-wiki.js
06数据库模型
db/schema.sql 是唯一可信的数据库结构来源，既定义表，也初始化默认配置。
配置与权限
system_config 保存初始化标记、管理员密码、站点标题、AI 配置、AI Wiki 快照等。
核心业务表
categories、bookmarks、deleted_bookmarks
统计与行为表
bookmark_visits、daily_analytics、search_logs、user_sessions
缓存与分析表
popularity_cache、category_analytics
07认证与安全
这个项目是后台系统，所以认证和默认密码逻辑是非常关键的一条主线。
密码优先级
ADMIN_PASSWORD 环境变量 → D1 中 system_config.admin_password → schema 默认 admin123
登录逻辑
/api/auth/login 登录后发放 JWT，默认有效期 24 小时。
前端守卫
auth.js 会做 token 校验缓存、失效后重定向回 /login.html。
全局保护
Pages Functions 中间件处理 CORS、安全头和部分可疑访问拦截。
08AI Wiki 能力
这是项目里比较有辨识度的增强点，不只是“书签 CRUD”，而是会自动整理书签知识结构。
pages/functions/api/wiki/index.js
负责读取 / 生成 AI Wiki 快照，并将结果写回 system_config。
pages/functions/utils/ai-wiki.js
执行数据去重、分类聚合、消息构建、调用 AI 和 fallback 生成逻辑。
public/js/wiki-view.js
前端提供本地视图 / AI 视图切换、命令面板、AI 自动整理按钮。
09运维与部署
项目的运行模型已经明显收敛到 Cloudflare 平台优先。
.github/workflows/frontend_pagefunction_deploy.yml
自动准备 Pages 项目和 D1 数据库，再部署 public/ 和 pages/functions/。
pages/wrangler.toml
本地开发时的 Pages 包装配置，生产由 GitHub Actions + Cloudflare API 接手。
scripts/setup-config.cjs / scripts/reset.cjs
生成本地配置、初始化或重置本地 D1 数据库。
本地开发命令
npm run config → npm run db:init:local / npm run reset → npm run dev
一句话判断： 这个项目的本质是一个“小而完整的 Cloudflare 后台系统”。 表面上是书签导航，但实际已经具备了 认证、 CRUD、 回收站、 统计分析、 备份、 AI 知识整理、 自动部署 这些典型后台能力。

主流程脑图

如果只想快速理解系统怎么跑，可以先看下面这条主流程。

Step 1 用户访问 进入 /login.html 或主页面。
Step 2 认证校验 前端 auth.js + 后端 /api/auth/* 完成 token 验证。
Step 3 业务操作 前端页面调用 /api/bookmarks/*、/api/system/*、/api/wiki 等接口。
Step 4 D1 读写 Functions 使用 BOOKMARKS_DB 读写分类、书签、统计、配置和快照。
Step 5 增强处理 需要时触发链接检查、备份、AI Wiki 生成、分析统计等辅助逻辑。
Step 6 部署交付 GitHub Actions 自动把静态页和 Functions 发布到 Cloudflare Pages。
